Why Stellar Works Without Custom Smart Contracts
Unlike Ethereum, where you must deploy custom smart contracts for every token or logic, Stellar has native operations built into the protocol . Think of these as pre-built, audited, network-level functions you just call.

Stellar Native Feature	What It Does	Investa Farm Use Case
Stellar Assets	Issue custom tokens without contracts 	Tokenize farm shares as "Farm Asset" tokens
Claimable Balances	Conditional payments with time locks 	Escrow for harvest payouts, investor commitments
Path Payments	Auto-convert between assets through the DEX 	Take M-Pesa KES → buy farm shares
Trustlines	Users opt-in to receive assets 	Investors explicitly choose to hold farm assets
2. The Specific Integrations You Need
2.1 Issuing Farm Assets (Without Smart Contracts)
Each farm you launch becomes a Stellar Asset. You don't deploy a contract – you just define the asset parameters .

What to tell Replit:

When a new farm is created on Investa Farm, automatically issue a unique Stellar Asset using the following parameters:

Asset Code: FARM_[farm_id] (e.g., FARM_101)

Issuer: Investa Farm's Stellar account public key

Flags: Authorization required (investors must be approved before receiving tokens)

Do not deploy any smart contract. Use the native Stellar Asset issuance via the stellar-sdk createAsset method. The total supply of the asset equals the number of farm shares (N).

2.2 Investor Trustlines (Opt-in to Receive)
Before an investor can receive farm share tokens, they must establish a trustline – a one-time operation that signals they trust your issuance .

What to tell Replit:

When an investor registers on Investa Farm, check if they have established a trustline to our platform's issuing account for all FARM_* assets. If not, prompt them to sign a changeTrust transaction using the Stellar SDK. This is a single approval that covers all future farm assets from our issuer.

2.3 Purchasing Farm Shares (Path Payments)
When an investor buys shares, they might pay in USD, KES, or even XLM. Stellar's path payments automatically convert through the built-in decentralized exchange .

What to tell Replit:

When an investor purchases farm shares:

Accept payment in any asset (KES via M-Pesa off-ramp, USDC, XLM)

Use Stellar's pathPaymentStrictReceive operation to automatically convert the payment into the exact farm asset

The conversion routes through the Stellar DEX – no manual swaps needed

If the path fails, fallback to a simple payment and handle conversion off-chain

2.4 Harvest Payouts as Claimable Balances (No Escrow Contracts)
This is where Stellar shines for your use case. Instead of building an escrow smart contract, use claimable balances – Stellar's native conditional payment system .

Example – Investor Harvest Payout with Time Lock:

javascript
// Create a claimable balance that the investor must claim within 30 days
// If not claimed, it returns to Investa Farm
const operation = Operation.createClaimableBalance({
  asset: usdcAsset,
  amount: investorPayoutAmount,
  claimants: [
    {
      destination: investorStellarAddress,
      predicate: { 
        beforeRelativeTime: "2592000"  // 30 days in seconds
      }
    },
    {
      destination: platformStellarAddress,  // fallback
      predicate: { 
        not: { beforeRelativeTime: "2592000" }  // after 30 days
      }
    }
  ]
});
What this gives you:

No smart contract deployment

Investors have 30 days to claim their payout

If they forget or their wallet is offline, funds automatically return to you

Claimable balances can be claimed by any qualifying claimant – perfect for multiple beneficiaries 

2.5 Commitment Watchlist (Time-Locked Purchases)
Your commitment/watchlist feature where investors lock funds for future purchase becomes a time-locked claimable balance .

What to tell Replit:

When an investor commits funds to the watchlist:

Create a claimable balance from the investor's account to themselves

Apply a not: { beforeRelativeTime: X } predicate – meaning they CANNOT claim before X days

Store the claimable balance ID in the database

When the lock period ends, the investor claims the balance (which contains the committed funds) and uses it to purchase shares

If they cancel early, the funds remain unclaimable until the time lock expires

4. What to Tell Replit (Complete Integration Spec)
Integrate Stellar network into Investa Farm using ONLY native operations – no custom smart contract deployment.

1. Setup & SDK

Install @stellar/stellar-sdk in the backend

Use Stellar Testnet for development, Mainnet for production

Create a platform Stellar account (issuer) with secret key stored in environment variables

2. Farm Asset Issuance (when farm launches)

Generate a unique asset code: FARM_[farm_id]

No contract deployment – Stellar Assets are native

The platform's account is the issuer

3. Investor Trustlines

Endpoint: POST /stellar/trustline

Creates a changeTrust transaction for the platform's issuing account

Investor signs once – all future FARM_* assets are automatically trusted

4. Share Purchase with Path Payment

Endpoint: POST /stellar/purchase/{farm_id}

Use pathPaymentStrictReceive to convert investor's payment (USDC/XLM/KES via off-ramp) into the farm asset

Automatically finds best route through Stellar DEX

5. Harvest Payouts as Claimable Balances

At harvest, for each investor: create a claimable balance with 30-day time lock

Claimants: [investor (30 days), platform (after 30 days)]

Store claimable balance IDs in the database for reconciliation

When investor claims, update internal ledger

6. Commitment Watchlist (Time-Locked Funds)

Endpoint: POST /stellar/commit

Create claimable balance from investor's account to themselves with NOT(BEFORE_RELATIVE_TIME(lock_days))

Funds are "locked" – investor cannot spend them until lock period expires

After expiry, investor claims and purchases shares

7. Important Rules

Never deploy custom Soroban contracts – use only native operations

All operations are standard Stellar transactions

The platform never holds investor private keys – investors sign with their own wallets or you use a custodial wallet (your choice per regulation)

Store all transaction hashes and claimable balance IDs in the database for audit
